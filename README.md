# AppInsight — 通用文本评论情感分析可视化架构

> **一套可复用的多维度评论数据可视化分析框架**，以 AWARE App 评论数据集为示范案例，展示从原始评论文本到 12 种交互式图表的完整分析管线。

## 🎯 核心价值：可视化架构，而非单一数据集

AppInsight 的本质不是「AWARE 数据集的可视化看板」，而是一套 **通用文本评论情感分析的可视化架构（Visualization Architecture）**。它的设计理念是：

> **只要你的数据包含「评论文本 + 情感标签 + 方面类别 + 评分」这几类字段，就能接入这套系统，自动生成同样丰富的多维度交互式分析界面。**

### 🔄 架构可扩展性

| 维度 | 说明 |
|------|------|
| **数据源无关** | 后端 `app.py` 的数据处理逻辑基于 Pandas DataFrame，只需替换数据文件和调整列名映射即可适配新数据集 |
| **领域可迁移** | 当前以 App 评论为案例，但同一架构可直接应用于：电商商品评价、酒店/餐厅点评、电影影评、社交媒体舆情、客服工单反馈等任意带标注的评论文本 |
| **方面类别自适应** | 玫瑰图、3D 气泡图、热力图等组件的类别轴动态读取数据中的唯一值，无需硬编码 |
| **图表组合即插即用** | 12 个图表组件独立封装，可根据新业务场景增删或重新组织页面布局 |

### 📦 适用数据格式

系统后端 `backend/app.py` 硬编码了以下 **7 个列名**，新数据集必须提供对应字段（CSV / JSON 均可）：

```
后端代码中的列名 ──→ 含义与要求
─────────────────────────────────────
sentiment        情感标签，值须为 'positive' 或 'negative'
category         方面类别（如：可用性、价格、性能…），字符串，用于玫瑰图/热力图/筛选器
rating           用户评分，数值型（通常 1-5），用于评分分析/排行榜/四象限/热力图
domain           领域分组（如：游戏、社交、工具…），字符串，用于雷达图
app              被评价实体名称（如 App 名/商品名/店铺名），字符串，用于河流图/排行/四象限
sentence         评论文本原始内容，字符串，仅用于「评论长度分析」图表
term             分词后的关键词/术语，字符串列表或字符串，仅用于「关键词词云」图表
```

> ⚠️ **关键点**：这 7 个列名是后端 Pandas 代码中直接引用的。接入新数据集时，**必须让数据的列名与之一致**（见下方两种方式）。

## ✨ 功能特点

### 🖥 界面设计
- **四页导航架构**：总览 / 情感 / 方面 / 排行，侧边栏切换，每页聚焦一类分析视角
- **全局筛选器**：顶部支持按情感类型（正面 / 负面 / 全部）和方面类别（12 类可选）筛选，所有图表实时联动更新
- **卡片式布局**：每个图表模块独立成卡，配有标题、说明文字和关键数据摘要
- **品牌标识**：AppInsight v2.0 统一品牌，页脚展示数据集来源与规模

### 📈 可视化能力（12 种图表 × 4 大分析层次）

#### 第一层：总体概览（总览页）
| 图表 | 图表类型 | 回答的问题 | 可复用性 |
|------|----------|-----------|---------|
| 情感仪表盘 | 仪表盘 + 环形图 | 整体正负面比例是多少？ | ✅ 任何有情感标签的数据 |
| 评分情感分析 | 折线图 + 组合柱状图 | 各星级的正面/负面率如何变化？ | ✅ 任何有评分+情感的数据 |
| 领域雷达图 | 雷达图 | 不同领域在各维度上表现差异？ | ✅ 替换领域分类即可 |
| 热门 App 河流图 | 堆叠面积图 | Top 实体的情感趋势如何演变？ | ✅ 替换实体名+时间戳即可 |

#### 第二层：情感深度（情感页）
| 图表 | 图表类型 | 回答的问题 | 可复用性 |
|------|----------|-----------|---------|
| 方面玫瑰图 | 南丁格尔玫瑰图 | 各方面类别的评论量分布？ | ✅ 动态读取类别 |
| 3D 气泡图 | 3D 气泡图 | 方面 × 情感的立体对比？ | ✅ 动态读取类别 |
| 评论长度分析 | 分组柱状图 + 折线 | 评论长度与情感有何关联？ | ✅ 任何文本数据通用 |
| 情感热力图 | 热力图 | 评分 × 方面的情感密度？ | ✅ 双轴动态 |

#### 第三层：方面挖掘（方面页）
| 图表 | 图表类型 | 回答的问题 | 可复用性 |
|------|----------|-----------|---------|
| 方面类别分布 | 南丁格尔玫瑰图 | 用户最关注哪些方面？ | ✅ 动态读取类别 |
| 热门 App 情感对比 | 条形对比图 | 高评论量实体的各维度表现？ | ✅ 替换实体即可 |

#### 第四层：实体排行（排行页）
| 图表 | 图表类型 | 回答的问题 | 可复用性 |
|------|----------|-----------|---------|
| 评分排行榜 | 水平条形图 | 哪些实体评分最高/最低？ | ✅ 任何有评分的数据 |
| 四象限分析 | 四象限散点图 | 评分 vs 讨论度的产品定位？ | ✅ 任何实体级数据 |

### 🔧 交互特性
- **悬停提示**：每个数据点均支持 Tooltip 查看详情
- **平滑动画**：图表渲染和数据切换时带有过渡动画
- **跨页联动**：筛选器变更后所有页面图表同步刷新
- **响应式布局**：适配桌面端浏览体验

## 💻 本地开发 & 接入自己的数据

### 前置要求

- Node.js ≥ 20
- Python ≥ 3.11
- Git

### 快速启动（使用默认 AWARE 数据）

```bash
# 1. 克隆项目
git clone https://github.com/YYJXX-lab/myWeb.git
cd myWeb

# 2. 启动后端
cd backend
pip install -r requirements.txt
python app.py          # 或 python render_start.py（使用 uvicorn）
# 后端服务将在 http://localhost:5000 启动

# 3. 启动前端（新终端）
cd frontend
npm install
npm run dev
# 前端服务将在 http://localhost:5173 启动
```

打开浏览器访问 `http://localhost:5173` 即可。

### 接入自定义数据集

接入新数据集的核心工作是 **让数据的列名与后端代码一致**。有两种方式：

#### 方式一：在加载数据时 rename（推荐，改动最小）

只改 `backend/app.py` 文件头部的数据加载部分：

```python
# ====== 只需修改这一处 ======
df = pd.read_csv('your_data.csv')  # 替换为你的数据文件路径

# 将你的原始列名映射到系统期望的 7 个列名
df.rename(columns={
    '你的情感列名':      'sentiment',   # 值须为 'positive' / 'negative'
    '你的方面类别列名':   'category',    # 字符串，如 "可用性"/"价格"
    '你的评分列名':      'rating',       # 数值，通常 1-5
    '你的领域分组列名':   'domain',       # 字符串，如 "游戏"/"社交"
    '你的实体名称列名':   'app',          # 字符串，如 App 名/商品名
    '你的评论文本列名':   'sentence',     # 字符串，用于评论长度分析
    '你的关键词列名':     'term',         # 字符串，用于词云图（可留空则词云不可用）
}, inplace=True)
# ============================

# 其余所有 API 路由中的 groupby / value_counts / pivot_table 逻辑
# 全部基于上面这 7 个标准列名工作，无需逐一改动
```

> ✅ **优点**：只需改文件开头几行，后续 ~500 行数据处理逻辑完全不用动。

#### 方式二：全局查找替换（适合列名已经接近的情况）

如果你的数据列名和系统列名差异不大，可以在 `backend/app.py` 中做全局替换：

```bash
# 例如：你的数据用 "app_name" 而系统用 "app"
sed -i "s/df\['app'\]/df['app_name']/g" backend/app.py
sed -i "s/\['sentiment'\]/['your_sent']/g" backend/app.py
# ... 以此类推，共 7 个列名
```

> ⚠️ **注意**：`app.py` 中这 7 个列名各被引用了 **10~60 次不等**（全文共约 80+ 处引用），方式一（rename）更安全、更可控。

#### 前端需要改吗？

**一般不需要。** 前端图表组件从 API 获取的是已聚合后的 JSON 数据（数值 + 标签字符串），不直接接触原始列名。只要后端 API 返回的 JSON 结构不变，前端会自动适应：
- 类别筛选器 → 自动读取 `category` 的唯一值
- 图表标签 → 自动显示你的实体名和类别名
- 筛选联动 → 同样基于后端返回的值工作

唯一可能需要调整的地方：如果新数据集的情感标签值不是 `positive/negative`（例如是 `pos/neg` 或 `1/0`），则需要在前端筛选器按钮的文本上做对应修改。

```bash
cd frontend
npm run build
# 构建产物输出到 frontend/dist 目录
```

## 📈 示范案例分析（AWARE 数据集）

> 以下结论来自当前线上演示所用的 **AWARE 数据集**，用于展示本套可视化架构所能产出的洞察深度。**换成你自己的数据集后，同样的图表将呈现属于你的洞察。**

基于 AWARE 数据集（11,321 条标注评论，12 个方面类别，5 个应用领域）：

1. **整体评价趋于平衡**：正面 50.1%，负面 49.9%
2. **三星用户是矛盾群体**：正面率仅 44.3%，最值得关注和挽回
3. **游戏 App 正面率最高**（52.4%），社交网络最低（44.7%）
4. **娱乐性是最大亮点**（正面率 69%），**可靠性是核心痛点**（负面率 73%）
5. **整体评价类最受关注**（约 3,000 条评论），远超其他方面类别
6. **长评论更倾向于负面表达**，用户在不满时更愿意详细阐述

## 🔧 云端部署说明

本项目采用 **前后端分离部署**方案：

### 前端（Vercel）
- 关联 GitHub 仓库，推送后自动构建部署
- `vercel.json` 配置构建命令和路由重写规则
- 构建完成后将 `frontend/dist` 静态托管

### 后端（Render）
- `render.yaml` 定义 Web Service 配置：
  - 运行时：Python（Singapore 区域）
  - 构建：`pip install -r backend/requirements.txt`
  - 启动：`gunicorn backend.app:app --workers 1 --threads 2 --timeout 120`
  - 健康检查：`/api/test`
- `wsgi.py` 作为 WSGI 入口，`render_start.py` 用于本地 uvicorn 开发

### 本地 ↔ 云端切换
- 前端 `http.ts` 中修改 API base URL 即可切换后端地址
- 后端 `app.py` 从环境变量读取端口，`host='0.0.0.0'` 适配云平台
